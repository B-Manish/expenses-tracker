# Deployment and Sync Guide

This guide explains how to keep the expenses tracker project in sync across local development, GitHub, Cloudflare Pages, Cloudflare Pages Functions, D1, environment variables, and production deployments.

The main idea is simple:

- Code and committed configuration are synced through Git.
- Secrets are synced through Cloudflare secrets and local `.dev.vars`.
- Database schema is synced through migration files.
- Production app updates happen through Cloudflare Pages deployments.
- Production data is not automatically copied to local unless you deliberately export/import it.

| Item | Value |
|---|---|
| Cloudflare Pages project | `expenses-tracker` |
| Production custom domain | `https://tracker.manishbatchu.com` |
| Build command | `npm run build` |
| Build output directory | `dist` |
| Cloudflare config file | `wrangler.toml` |
| D1 binding variable | `DB` |
| D1 database name in `wrangler.toml` | `expenses-db` |
| D1 database ID in `wrangler.toml` | `930109c1-d8de-4632-bc42-b7e3542351c5` |
| Local env file | `.dev.vars` |
| Migration folder | `migrations/` |

The binding name `DB` is important. The Functions code reads the database from `context.env.DB`, so changing the binding name will break API routes unless the code is updated too.

## What Goes Where

| Thing | Local source | Production source | Commit to Git? |
|---|---|---|---|
| React source code | `src/` | Built by Cloudflare Pages | Yes |
| Pages Functions code | `functions/` | Uploaded by Cloudflare Pages | Yes |
| Build config | `package.json`, `vite.config.*` if present | Cloudflare Pages build | Yes |
| Cloudflare Pages/D1 config | `wrangler.toml` | Cloudflare reads this during deploy | Yes |
| Database schema | `migrations/*.sql` | Applied to remote D1 | Yes |
| Local secrets | `.dev.vars` | Local Wrangler dev only | No |
| Production secrets | Cloudflare dashboard secrets | Cloudflare Pages runtime | No |
| Production data | Remote D1 database | Remote D1 database | No |
| Build output | `dist/` | Generated by build | No |
| Wrangler local state | `.wrangler/` | Local only | No |

## Git Sync

Git syncs code, migration files, and safe configuration.

Use this normal flow:

```bash
git status
git add .
git commit -m "Describe the change"
git push origin main
```

Cloudflare Pages is connected to the GitHub repo. Pushing to the production branch, currently `main`, triggers a new deployment.

Before pushing, usually run:

```bash
npm run build
```

If the change touches lint-sensitive code, also run:

```bash
npm run lint
```

## Environment Variables and Secrets

This app uses these runtime secrets:

| Name | Purpose | Should be secret? |
|---|---|---|
| `APP_PASSWORD` | Password used to log in to the app | Yes |
| `SESSION_SECRET` | Secret used to sign session cookies | Yes |
| `SMS_INGEST_TOKEN` | Restricted bearer token used by the iPhone Shortcut ingestion endpoint | Yes |

For local development, put them in `.dev.vars`:

```env
APP_PASSWORD=your-local-password
SESSION_SECRET=your-local-long-random-secret
SMS_INGEST_TOKEN=your-local-random-token-of-at-least-32-characters
```

Do not commit `.dev.vars`. It is already ignored by `.gitignore`.

For production, set them in Cloudflare:

1. Open Cloudflare dashboard.
2. Go to `Workers & Pages`.
3. Open `expenses-tracker`.
4. Open `Settings`.
5. Select environment `Production`.
6. Go to `Variables and Secrets`.
7. Add or edit these values as `Secret`, not plaintext:
   - `APP_PASSWORD`
   - `SESSION_SECRET`
   - `SMS_INGEST_TOKEN`

Because this project uses `wrangler.toml`, Cloudflare may show a message saying environment variables are managed through `wrangler.toml` and only encrypted secrets can be managed through the dashboard. That is expected. For this app, `APP_PASSWORD` and `SESSION_SECRET` should be dashboard secrets.

After changing production secrets, trigger a fresh deployment so the live Functions pick up the new runtime configuration.

## D1 Binding Sync

The app expects one D1 binding:

```toml
[[d1_databases]]
binding = "DB"
database_name = "expenses-db"
database_id = "930109c1-d8de-4632-bc42-b7e3542351c5"
```

This lives in `wrangler.toml` and should be committed.

In Cloudflare Pages settings, the binding should match:

| Field | Value |
|---|---|
| Binding type | D1 database |
| Variable name | `DB` |
| Database | `expenses-db` |

If the binding name is not `DB`, API routes that use the database will fail.

If `wrangler.toml` contains a placeholder database ID, deployments can fail with an error like:

```text
Invalid database UUID (REPLACE_WITH_CLOUDFLARE_D1_DATABASE_ID)
```

Fix that by replacing the placeholder with the real D1 database ID.

## Database Schema Sync

Local and production databases do not automatically share schema changes. Schema sync is done with migration files in `migrations/`.

For this project, the initial migration is:

```text
migrations/0001_initial.sql
```

That migration creates the core tables:

- `categories`
- `payment_methods`
- `transactions`
- `settings`

It also inserts default categories, payment methods, and settings.

Additional migrations include:

- `migrations/0018_mcp_tokens.sql` — creates the `mcp_tokens` table for per-user MCP authentication tokens.

### Apply Migrations Locally

Run this after creating or changing migration files:

```bash
npx wrangler d1 migrations apply DB --local
```

### Apply Migrations Remotely

Run this before or during production deployment whenever production needs a schema change:

```bash
npx wrangler d1 migrations apply DB --remote
```

If Wrangler cannot access D1, check that you are logged in or that `CLOUDFLARE_API_TOKEN` has D1 permissions.

### Verify Remote Tables

In the Cloudflare D1 SQL console, run:

```sql
SELECT name
FROM sqlite_master
WHERE type = 'table'
ORDER BY name;
```

A healthy production database should show app tables such as:

- `categories`
- `payment_methods`
- `settings`
- `transactions`

If it only shows `_cf_KV` or `_cf_kv`, the app schema has not been applied to that database yet.

## Data Sync

Schema and data are different.

Migration files keep schema in sync. They should not be used as a general way to copy real transaction data between environments.

Use this rule:

| Need | Recommended action |
|---|---|
| Add or change tables/indexes/default seed data | Create a migration |
| Add normal expenses/income | Use the app UI or API |
| Inspect production data | Use D1 SQL console or Wrangler query |
| Copy production data to local | Export/import intentionally |
| Copy local test data to production | Avoid unless it is deliberate seed data |

Production D1 is the source of truth for real user data.

## Local Development Flow

Use this when working locally:

```bash
npm install
npm run build
npx wrangler d1 migrations apply DB --local
npx wrangler pages dev dist
```

If local Pages dev does not pick up the D1 binding automatically, run with an explicit binding:

```bash
npx wrangler pages dev dist --d1 DB=930109c1-d8de-4632-bc42-b7e3542351c5
```

For frontend-only work, `npm run dev` is useful, but it does not fully match Cloudflare Pages Functions behavior. For API and auth testing, prefer Wrangler Pages dev.

## Normal Deployment Flow

Use this flow for regular code-only changes:

```bash
npm run build
git status
git add .
git commit -m "Describe the change"
git push origin main
```

Then check Cloudflare:

1. Open `Workers & Pages`.
2. Open `expenses-tracker`.
3. Go to `Deployments`.
4. Open the newest deployment.
5. Confirm build and Functions publish successfully.

## Deployment Flow With Database Changes

Use this flow when a change includes new migrations:

```bash
npm run build
npx wrangler d1 migrations apply DB --local
npx wrangler d1 migrations apply DB --remote
git status
git add .
git commit -m "Describe the schema/app change"
git push origin main
```

The remote migration should usually be applied before testing the new production deployment if the new code expects the new schema.

For destructive schema changes, pause and take a backup/export first. Avoid dropping tables or columns in production unless you are sure the app no longer needs them.

## Triggering a Redeploy

Cloudflare Pages usually deploys automatically when you push to `main`.

If you need to redeploy without code changes, create an empty commit:

```bash
git commit --allow-empty -m "Trigger Cloudflare redeploy"
git push origin main
```

Then watch the newest deployment in Cloudflare under the `Deployments` tab.

## Production Verification Checklist

After each production deployment, verify:

- The latest Cloudflare deployment succeeded.
- There is no `Invalid database UUID` error.
- Functions published successfully.
- `https://tracker.manishbatchu.com/api/health` returns success.
- Login works with the production `APP_PASSWORD`.
- Logged-out protected routes return an auth error instead of data.
- Dashboard loads without `Internal server error`.
- Add expense works.
- Add income works.
- Edit transaction works.
- Delete transaction works.
- Categories load.
- Payment methods load.
- Settings load and save.
- A token generated in Settings → MCP Access authenticates `POST /mcp` (`initialize` returns a result).
- A revoked or unknown token returns 401 from `POST /mcp`.

Useful URLs:

```text
https://tracker.manishbatchu.com/
https://tracker.manishbatchu.com/api/health
https://tracker.manishbatchu.com/api/auth/me
https://tracker.manishbatchu.com/api/stats
```

`/api/auth/me` may return an authentication-required response when logged out. That is fine. It should not return `Authentication is not configured`.

## Common Problems and Fixes

### `Authentication is not configured`

Meaning:

The deployed Function cannot see `APP_PASSWORD` and/or `SESSION_SECRET`.

Fix:

1. Open Cloudflare Pages project settings.
2. Select `Production`.
3. Confirm `APP_PASSWORD` exists as a secret.
4. Confirm `SESSION_SECRET` exists as a secret.
5. Redeploy.

### `Invalid database UUID`

Meaning:

`wrangler.toml` has a bad or placeholder D1 database ID.

Fix:

1. Open Cloudflare D1 database details.
2. Copy the real database ID.
3. Update `wrangler.toml`.
4. Commit and push.

### `/api/stats` returns `Internal server error`

Likely meaning:

The app reached D1, but the database does not have the required tables.

Fix:

1. In D1 SQL console, run:

   ```sql
   SELECT name
   FROM sqlite_master
   WHERE type = 'table'
   ORDER BY name;
   ```

2. If app tables are missing, apply migrations:

   ```bash
   npx wrangler d1 migrations apply DB --remote
   ```

3. Test `/api/stats` again after logging in.

### D1 Binding Is Missing

Meaning:

`context.env.DB` is not available to Pages Functions.

Fix:

1. Confirm `wrangler.toml` has `binding = "DB"`.
2. Confirm Cloudflare Pages settings has a D1 binding named `DB`.
3. Redeploy.

### Wrangler D1 Commands Fail With Auth Errors

Meaning:

Wrangler is not logged in, or the API token does not have permission to list/query/migrate D1.

Fix:

Use one of these approaches:

```bash
npx wrangler login
```

or create/update a Cloudflare API token with permissions for:

- Account access for the correct Cloudflare account
- D1 read/write access
- Workers/Pages access if you also deploy through Wrangler

Store local-only tokens in `.dev.vars` or your shell environment. Do not commit them.

## Safe Secret Rotation

To rotate the app password:

1. Update `APP_PASSWORD` in Cloudflare Production secrets.
2. Update local `.dev.vars` if you want the same password locally.
3. Redeploy.
4. Log in with the new password.

To rotate the session secret:

1. Update `SESSION_SECRET` in Cloudflare Production secrets.
2. Update local `.dev.vars` if needed.
3. Redeploy.
4. Log in again.

Changing `SESSION_SECRET` invalidates existing sessions, which is expected.

## Quick Golden Path

For a normal app change:

```bash
npm run build
git add .
git commit -m "Update app"
git push origin main
```

For a schema change:

```bash
npm run build
npx wrangler d1 migrations apply DB --local
npx wrangler d1 migrations apply DB --remote
git add .
git commit -m "Update database schema"
git push origin main
```

For a secret change:

```text
Update Cloudflare Production secret -> redeploy -> verify login/API
```

For a production API error:

```text
Check deployment logs -> check secrets -> check D1 binding -> check D1 tables -> check migrations
```
